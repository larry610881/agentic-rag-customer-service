# 為什麼要做 AI 智慧客服系統

## 一、AI 智慧客服的產業趨勢

### 從「聊天機器人」到「AI Agent」

2026 年，AI 客服已經從過去的腳本式問答機器人，進化到能理解上下文、主動執行動作的 AI Agent。這不只是技術升級，而是客服型態的根本轉變。

| 世代 | 技術 | 能力 | 代表 |
|------|------|------|------|
| 第一代 | 關鍵字比對 | 固定問答，命中率低 | 傳統 FAQ Bot |
| 第二代 | RAG（檢索增強生成） | 能根據知識庫回答，但只能「說」 | LumineAI 等基本 RAG 方案 |
| 第三代 | AI Agent（ReAct） | 能理解意圖、呼叫工具、幫客戶「做事」 | 本系統 |

### 市場數據

- **80% 的客服組織**已導入或計畫導入 AI 來提升客服生產力與客戶滿意度（Gartner, 2025）
- **78% 的企業**在至少一個業務功能中使用 AI，較前一年的 55% 大幅成長（McKinsey State of AI Global Survey, 2025）
- Gartner 預估，對話式 AI 將在 2026 年為客服營運**節省 800 億美元**成本
- 客戶期待正在改變：不只要「回答問題」，更要「解決問題」

### 智慧客服的核心價值

真正的「智慧」不是能回答問題，而是能幫客戶做事：

- 查商品庫存、比較價格
- 查課程空位、預約報名
- 查訂單狀態、處理退換貨
- 根據對話內容主動推薦

如果 AI 客服只能聊天，那跟 Google 搜尋沒有本質差異。能幫客戶執行動作，才是 AI Agent 的價值所在。

## 二、LumineAI 方案的侷限

我們在評估階段試用了 LumineAI 的 AI 客服方案，發現以下問題：

### 功能侷限

| 面向 | LumineAI 現況 | 問題 |
|------|-------------|------|
| RAG 架構 | 基本切片，沒有進階處理 | 文件只做單純切割，無 metadata 過濾、無語意分段 |
| Agent 模式 | Router（路由分流） | 只能判斷意圖分流，不能推理和執行多步驟動作 |
| 工具能力 | Web 搜尋 + 圖片生成（Nano Banan） | 跟客服業務完全無關，無法查商品、訂預約 |
| 客製化 | 有限 | 可自訂 Bot Prompt，但無法串接自有業務系統 |
| Prompt Engineering | 我方負責 | 付了 5 萬月費，但 Prompt 調校還是我方自己做 — 等於付錢買基礎設施，核心品質還是靠自己 |
| 可觀測性 | 無反饋機制 | 回答不好時，無法判斷是 RAG 撈到的原始資料有問題、還是 Prompt 寫得不好、還是 LLM 幻覺 — 沒辦法系統性改善品質 |

### 本質問題

LumineAI 提供的是**第二代 RAG 問答機器人**，不是第三代 AI Agent。它能回答「你們有賣什麼？」，但不能幫客戶「幫我查這個商品還有沒有貨」然後直接回覆即時庫存。

更關鍵的是：即使用了 LumineAI，**Prompt Engineering 的工作量不會減少**。RAG 系統中 Prompt 對輸出品質的影響占比極大，這部分無論用哪套系統都是我方負責。差別在於，LumineAI 沒有提供任何診斷工具讓你知道問題出在哪裡。

### 結論

就算使用市場上最貴的 LLM 模型（Claude Opus 4.6），18M tokens 的純 API 成本也只有 3,510 TWD — LumineAI 收 50,000 TWD，**價差達 14 倍**。

而電商客服實務上使用「客服推薦級」模型（如 GPT-5-mini、Gemini 2.5 Flash）即可滿足需求，成本不到 250 TWD — **價差超過 200 倍**。

LumineAI 方案 = **功能停在第二代 + 價格收第四代的錢**。

這就是我們決定自建的原因：**品質可控、成本可控、功能可擴展**。
