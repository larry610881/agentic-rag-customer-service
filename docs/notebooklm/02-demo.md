# Demo 對比：LumineAI vs 自建 AI 客服

## 一、兩邊系統簡介

### LumineAI

| 項目 | 說明 |
|------|------|
| 定位 | SaaS AI 客服機器人平台 |
| 架構 | 基本 RAG（單純文件切片） |
| Agent 模式 | Router（意圖分流） |
| 可用工具 | Web 搜尋、Nano Banan 圖片生成 |
| Prompt 客製化 | 可對每個 Bot 自訂 Prompt |
| 業務工具串接 | 無（無法串接客戶自有系統） |
| 定價 | 18M tokens / 50,000 TWD |

### 自建 AI 客服系統

| 項目 | 說明 |
|------|------|
| 定位 | 多租戶 AI Agent 客服平台 |
| 架構 | 進階 RAG + LangGraph Agent |
| Agent 模式 | Router（分流）+ ReAct（推理 + 執行動作） |
| 可用工具 | 可依客戶業務自訂 — 查商品、查課程、訂預約等 |
| Prompt 客製化 | 完全自訂 System Prompt + 動態變數注入 |
| 業務工具串接 | 透過 MCP Tool 串接客戶自有資料庫與 API |
| LLM 模型 | 可切換 — OpenAI / Claude / Gemini / DeepSeek |
| 多租戶 | 支援，一套系統服務多家客戶，資料完全隔離 |
| 定價 | 按實際 LLM API 用量 + 基礎設施，成本透明 |

## 二、Demo 流程設計

> Demo 目標：讓部長在 10 分鐘內感受兩套系統的差距。
> 方式：Live Demo（準備錄影備案）。
> 公平對比條件：兩邊使用**相同的 Prompt** 和**相同的 LLM 模型（GPT-5.1）**，排除模型差異因素，純粹比較系統能力。

### 階段 1：基本問答對比（Router 模式）

> 目的：展示兩邊都能做到的事 — 基於知識庫的問答。

**Demo 腳本（以窩廚房料理課程為例）：**

| 步驟 | 使用者提問 | LumineAI 預期回答 | 自建系統預期回答 |
|------|----------|----------------|---------------|
| 1 | 「你們有什麼料理課程？」 | 從知識庫摘要回答課程列表 | 從知識庫摘要回答課程列表 |
| 2 | 「親子課程有哪些？」 | 從知識庫過濾回答 | 從知識庫過濾回答 |
| 3 | 「價格大概多少？」 | 從知識庫摘要回答價格範圍 | 從知識庫摘要回答價格範圍 |

**這個階段兩邊表現差不多** — 證明基本 RAG 問答不是門檻，兩邊都做得到。

### 階段 2：進階功能對比（ReAct Agent 模式）

> 目的：展示 LumineAI 做不到、我們能做到的事 — 幫客戶「做事」。

**Demo 腳本：**

| 步驟 | 使用者提問 | LumineAI 預期回答 | 自建系統預期回答 |
|------|----------|----------------|---------------|
| 1 | 「下週六的親子課程還有名額嗎？」 | 無法回答（知識庫沒有即時庫存資料） | 呼叫 `query_courses` Tool → 即時查詢 DB → 回覆「XX 課程還有 3 個名額」 |
| 2 | 「幫我查 2000 元以下的低溫宅配商品」 | 無法精準回答（無法查 DB） | 呼叫 `query_products` Tool → 條件篩選 → 回覆商品清單含價格 |
| 3 | 「我想報名週六那堂課」 | 無法執行（沒有預約工具） | 呼叫預約 Tool → 完成報名 → 回覆「已幫您預約成功」 |
| 4 | 「幫我查一下冷凍的人氣商品」 | 無法精準回答 | 呼叫 `query_products(ship_type=3)` → 回覆冷凍商品推薦清單 |

**這個階段是殺手鐧** — LumineAI 只能說「請聯繫客服」，我們直接幫客戶完成。

### 階段 3：補充展示（可選）

| 功能 | 說明 |
|------|------|
| 多租戶切換 | 展示同一套系統切換不同租戶，知識庫和設定完全隔離 |
| LLM 模型切換 | 展示同一個問題用不同模型回答，品質差異不大但成本差很大 |
| 知識庫管理 | 展示上傳文件 → 自動分塊向量化 → 立即可被 AI 檢索 |

## 三、Demo 注意事項

### 事前準備

- 確認網路環境穩定
- 準備好窩廚房的測試資料（課程 + 商品）
- 錄製一次完整 Demo 作為備案影片
- 準備 2-3 個「必殺問題」確保 Demo 順暢

### 風險控管

| 風險 | 備案 |
|------|------|
| Live Demo 當機 | 切換到預錄影片 |
| LLM API 回應慢 | 提前暖機，或切換到較快的模型（如 GPT-5-mini） |
| 回答品質不佳 | 事先測過的問題為主，避免即興提問 |

### 預期效果

部長看完 Demo 後應該能得出結論：
1. **基本問答兩邊都行** — 這不是技術門檻
2. **我們能做到 LumineAI 做不到的事** — 即時查詢、執行動作
3. **接下來看成本** — 自然過渡到第三章成本比對
